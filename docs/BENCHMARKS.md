## 1. Benchmark Philosophy

The Horus Benchmark Suite (HBS) is structured around three verifiable claims
that underpin the NFE architecture:

1. **Outlier isolation claim:** Per-value exponent encoding prevents a single
   high-magnitude value from degrading precision for the remaining operands in
   the same computation block.

2. **Throughput claim:** The 13-bit word width and single-cycle MAC path
   maintain accumulator throughput competitive with or exceeding 8-bit integer
   formats in attention-projection kernels.

3. **Quantization error claim:** The 6-bit mantissa (plus hidden bit) keeps
   MUL relative error below 1.5625% — the theoretical bound — while ADD and
   SUB remain lossless across the full 64-stop exponent ladder.

Every benchmark in this suite is designed to independently stress-test one of
these three claims. No benchmark conflates them.

**Reproducibility standard:** Each benchmark specifies (a) the input generator
or test-vector source, (b) the execution environment, (c) the reference
implementation for comparison (FP64), and (d) the pass/fail criterion. A
benchmark is not considered verified until all four are published alongside
the results.

## 3. HBS-1: 128×128 GEMM Kernel

### Objective

Measure the accuracy loss and compute-cycle efficiency of a 128×128 matrix
multiply (GEMM) when operands are quantized from FP64 to NFE v3, compared to
INT8, FP8 (E4M3, E5M2), and MXFP8 (E4M3 block-32).

A 128×128 GEMM is the smallest kernel that:
- Fully utilizes a 4×4 systolic array across 32 pipeline-fill cycles.
- Contains enough rows to expose intra-block outlier interactions in block-scaled formats.
- Maps directly to one attention head projection slice (Q/K/V linear layer,
  sequence stride = 128).

### Computation model

```
C[128×128]  =  A[128×128]  ×  B[128×128]
```

Each output element `C[i,j]` is the inner product of row `i` of A with
column `j` of B — a 128-element dot product accumulating into a 32-bit
`accum_reg`. With the 4×4 Horus systolic array, this requires 32 tile
operations per output element.

### Cycle-accurate simulation methodology

**Execution environment:** Icarus Verilog simulation using
`horus_mesh_top.v` (N×N tiled mesh) with `MESH_SIZE = 4` and
`FLIT_W = 17` (4-bit XY header + 13-bit NFE payload).

**Stimulus generation:**
1. Test matrices A and B are generated by `horus_nfe_analysis.py` in three
   variants: (a) uniform random in [0.1, 1.0], (b) outlier-heavy (1% of
   values at 10× the mean), (c) near-zero (all values in [0.001, 0.05]).
2. Each element is encoded to the target format (NFE, INT8, FP8, MXFP8)
   using the respective reference encoder.
3. The quantized matrices are injected into the mesh via West boundary flits
   (`west_din_flat`, row-major order), one 13-bit operand per flit.
4. Tile results are collected from `result_flat` after pipeline drain.

**Reference:** FP64 GEMM of the original (pre-quantization) matrices.

**Error metric:**
```
Frobenius Relative Error  =  ‖C_quantized − C_fp64‖_F  /  ‖C_fp64‖_F
```

**Throughput metric:**
```
Effective MACs/cycle  =  (128 × 128 × 128 MACs)  /  (total simulation cycles)
```

## 4. HBS-2: Adversarial Outlier Block

### Status: ✅ Complete — reproducible from `horus_nfe_analysis.py`

**Test vector:** 32-element block, 1 outlier at 10.0, 31 values uniformly
in [0.001, 0.031].

| Format | Mean Rel. Error | FTZ floor hits | Outlier saturates? |
|--------|:--------------:|:--------------:|:------------------:|
| **NFE v3 (Bias-32)** | **0.31%** | **0 / 32** | No |
| MXFP8 E4M3 (block-32) | 0.69% | 0 / 32 | No (block-scaled) |
| E4M3 per-element | 7.15% | 0 / 32 | No |
| FP16 (baseline) | <0.01% | 0 / 32 | No |
| NFE v1/v2 (deprecated) | 40.59% | 7 / 32 | No |

> **NOTE on v1/v2 baseline:** An earlier iteration of the script modelled the
> pre-Bias-32 zero-biased exponent encoding (`value = 2^E × M/64`), which
> crushes all values below 1/64 = 0.015625 to zero (FTZ) and produces Ghost
> Zero in MUL. That version showed 40.59% mean error and is retained as
> Section 0 of the script for full transparency. The v3 Bias-32 encoding
> (current hardware) eliminates both defects.

**Reproduce:**

```bash
python3 horus_nfe_analysis.py
```

The adversarial block result is printed in **Section 1** of the script output
(`main()` function). Section 0 (`show_v1_baseline()`) shows the deprecated
v1/v2 result for comparison. Section 2 (`analyze_add_frac_delta()`) documents
the ADD_FRAC delta precision floor — a known limitation of the ADD operation.

## 5. HBS-3: Log-Uniform Quantization Sweep

### Status: ✅ Complete — reproducible from `horus_nfe_analysis.py`

Across all 64 octaves (actual_E = −32 to +31), 1,000 uniform-random FP64
values per octave are encoded and decoded through NFE v3. The relative error
at each octave boundary is measured.

| Metric | Result | Expected (theory) |
|--------|--------|--------------------|
| Mean relative error per octave | ~0.27% | ≤ 0.78% |
| Max relative error (all octaves) | ≤ 0.78% | = 1/128 (one LSB step) |
| Error variance across octaves | Negligible | 0 (log-uniform property) |

The constant per-octave error is the defining property of **log-uniform
quantization**: resolution scales with magnitude, matching the empirical
distribution of neural network weights and activations.

## 6. HBS-4: Transformer Attention Projection

### Status: 🔲 Defined / not yet executed

**Target model:** Llama-3 8B (Meta AI, 32 attention heads, head dimension 128)

**Layer under test:** Q, K, V, and O projection matrices in one attention head.

**Methodology:**
1. Extract a representative activation tensor from a Llama-3 8B inference run
   on a standard prompt (e.g., MMLU 5-shot validation set, first 100 samples).
2. Quantize the activation and weight tensors to each candidate format.
3. Compute the attention score matrix `softmax(QKᵀ / √d)` using quantized
   operands.
4. Measure **attention output error** relative to FP32 reference:
   ```
   ‖O_quantized − O_fp32‖_F / ‖O_fp32‖_F
   ```

## 9. Benchmark Execution Guide

### 9.1 Reproducible today

```bash
# C-model: 10M-iteration error proof
gcc -O2 -o horus_sim horus_sim.c -lm && ./horus_sim

# Python: adversarial outlier, log-uniform sweep, format-width analysis
python3 horus_nfe_analysis.py

# Simulation harnesses (requires Icarus Verilog ≥ 11)
iverilog -Wall -o sim_mesh \
    tb_horus_mesh_top.v horus_mesh_top.v horus_router.v \
    horus_system.v horus_nfe.v horus_pgate_ctrl.v
vvp sim_mesh
```
